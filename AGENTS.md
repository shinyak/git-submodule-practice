# 学習用ルール: git submodule プラクティス

このリポジトリは「git submodule を再現環境で安全に体験し、運用面まで使えるようになる」ための学習用領域として使う。

## 1. 目的

- git submodule の仕組み（gitlink / .gitmodules / commit 参照）の理解
- `add / status / init / update / sync / deinit / foreach / absorbgitdirs / submodule absorbgitdirs` などの実戦操作
- サブモジュールを含むワークフロー（更新、追跡、ブランチ運用、バージョン固定、トラブルシューティング）をハンズオンで習得
- 実運用で使える「手順」「判断基準」を体得する

## 2. 対象者の前提（このリポジトリでの前提）

- 既知:
  - merge と rebase の違い
  - `--ff` や fast-forward の基本
- 追加で前提化:
  - `git status`, `git log`, `git diff`, `git add/commit`, `git remote -v` などの基本コマンド
- 学習時の想定条件:
  - コマンドは原則ローカルで再現し、必要時だけネットワーク（`origin` 取得）に触れる
  - 変更は最小単位で小分けし、`commit` メッセージは意味が通るものを使う

## 3. このリポジトリでの進行ルール

- 再開時は次を順番に実行して判定する。  
  **`Phase1` は明示的に「完了」ログを残すまで決してスキップしない**。

  1. `git status`
  2. `git log --oneline -n 8`
  3. `git submodule status`
  4. `LEARNING_LOG.md` の最新エントリ確認

  判定基準（再開時ゲート）:

  1. まず `LEARNING_LOG.md` の最新エントリの「現状 / ステータス / 次」を確認する。
  2. 最新エントリで `現状` が `Phase1` 未完了または `進行中` を示している場合は、そのフェーズを再開する。
  3. 最新エントリで `次` が `Phase1` を明示している場合は `Phase1` から再開する。
  4. それらに該当しない場合は `git submodule status` を補助判定に使う。

  `git submodule status` 補助判定:

  - `git submodule status` が空: サブモジュール未登録。`Phase1`（基礎理解）完了後に `Phase2` へ
  - 出力があり、`-` が先頭にある: サブモジュール未初期化。`git submodule init / update` を確認し `Phase2` へ
  - 出力があり、`+` が先頭にある: 親が参照しているコミットと実体がズレ。`Phase3`（更新と追従）へ
  - 出力に `U` や `M`（dirty）表示: `Phase5`（障害対応）へ
  - 出力が clean (`<hash> path`) のみ: 現在のフェーズ継続または次の設計検証。`LEARNING_LOG.md` の最新エントリの「ステータス」で分岐

### 進行時の最小条件（Phaseの誤移行防止）

- `Phase1` は次を実行して完了扱いにする。
  - `.gitmodules` / `gitdir` / `.git/config` がサブモジュールと関係する前提知識を説明できる
  - `git submodule status` の `-` `+` `U` `M` の意味を説明できる
  - サブモジュール未登録状態の意味を説明できる
- 上記のどれかが未実施なら、次ステップは必ず `Phase1` 再開。

  失敗が出た場合は、再実験前に `LEARNING_LOG.md` の同日エントリへ「再現手順 / 原因推定 / 対処 / 観測」を1行ずつ残すこと。
- 学習セッションは 1 つのステップ（Phase）単位で行う。
- 進行はハンズオン前提で、学習者は別ターミナルで実コマンドを実行しながら学習を進める。
- サポート側は各手順で「目的・背景・実行コマンド・確認ポイント」を提示し、1手順ずつ進める。
- 学習者が途中で質問した場合は、質問対応を優先して回答し、回答後に中断していた学習シーケンスに戻す。
- サポート側は各セッションで進捗を逐次更新し、次の3つを都度明示する。
  - `現状: 今何をやっているか`
  - `達成: 直近で確認済みの成果`
  - `次: 次に実施するアクション`

- 各セッションは必ず次を実施する:
  1. 直前状態確認（`git status`, `git log --oneline -n 8`, `git submodule status`）
  2. 学習目標を 1 行で宣言
  3. 実験（コマンド実行）
  4. 結果の検証（`git diff`, `git log`, `git submodule status`, 確認コマンド）
  5. 学んだことの要約（1〜2 行）
  6. 決め手（次に進む条件）を明記
  - 進捗更新はこれらの実施内容と対応づけて、セッション中に随時実施する。

- 失敗は前提。再現手順・原因推定・対処を必ず `学習ログ` に残す。
- 迷ったときは **submodule 側で commit を更新してから親側が参照を更新**する流れを優先して考える。
  4. 学習ログは次テンプレートに従って、セッション開始時と終了時の両方で記録する。

### ハンズオン形式の基本形（推奨）

- 提示順（毎回同じ型）:
  1. `今回やること`
  2. `なぜ必要か（背景）`
  3. `実行すべきコマンド`
  4. `そのコマンドが何をしたか` の解説
  5. `結果の見取り図`（何が変わる想定か）
  6. `次の一手`（次に進む条件）

- 進捗判定は「コマンドを実際に実行して確認できたか」で判断する。
- 進行順は、原則として以下の順を維持する:
  - Step 1: 状態確認
  - Step 2: 背景理解
  - Step 3: 実行
  - Step 4: 検証
  - Step 5: 振り返り

## 4. 進捗管理ルール

- AGENTS に従った進捗は別ファイルの [LEARNING_LOG.md](/Users/skanno/project/shinyak/git-submodule-practice/LEARNING_LOG.md) に追記する。
- 再開時はまず `LEARNING_LOG.md` の最新エントリから「次に何をやるか」を引き継ぐ。
- 進捗は以下 5 段階:
  - `未着手`
  - `進行中`
  - `完了`
  - `再確認必要`
  - `運用反映済み`
- 重要な実験結果は「成功 / 失敗 + 何を観測できたか」を1行で残す。

### フォーマット（追記テンプレ）

- 日付:
  - `日時: 2026-03-15`
  - `フェーズ:`
  - `目的:`
  - `ステータス:`
  - `実行コマンド:`
  - `観測結果:`（成功 or 失敗 + 何を確認できたか）
  - `現状:`
  - `達成:`
  - `次:`
  - `次アクション:`

#### 追加: 再開時の最短判定テンプレ

- `現状` / `達成` / `次` はログ本文にも残すと再開が楽。
- `現状` には再開時ゲートの判定結果を1行で書く（例: `-` のみ / `+`あり / dirty）。
- `次` には「なぜそのフェーズへ進むのか」を明確に1行で書く。
- `観測結果` は最低1行で「成功 or 失敗 + 何を確認できたか」を記録。

## 5. 学習プラン（実践中心）

### Phase 1: サブモジュールの基礎理解

- サブモジュールの構成要素（`.gitmodules` / gitlink）を観察
- `git submodule status` の意味と表示差分を理解
- 既存リポジトリをサブモジュールとして扱う意味を確認

### Phase 2: 追加から初期化まで

- 空の親リポジトリにローカルの子リポジトリを submodule で追加
- `git submodule add`, `git submodule init`, `git submodule update` の違い
- `--depth` / `--branch` の挙動を比較して理解

### Phase 3: 更新と追従

- 子でコミットを進めた後、親側で参照を更新する一連の流れ
- `git submodule update --remote` と明示 `git commit`（固定 commit）を比較
- `git fetch`, `git checkout`, `git pull` の責務分離

### Phase 4: 分岐・運用ルール

- `submodule.<name>.update` や `--merge` / `--rebase` の取り扱い実感
- 共有ブランチでの運用を想定した更新手順を作る
- チーム内での `freeze`（固定コミット運用）と `rolling`（追従）ポリシーを比較

### Phase 5: 障害対応

- よく起きる事故:
  - 親で submodule が dirty（未コミット）
  - `git submodule status` の `-`/`+`/`U` 系表記の意味不一致
  - `--recursive` 忘れ
  - `.gitmodules` 変更後の同期漏れ
- 事故対応を `再現 → 原因切り分け → 修正 → 再確認` で運用

## 6. 今回の学習の実行基準

- すべての Phase を通じて、次が満たせたら「submodule 実戦OK」:
  - 親リポジトリから子リポジトリの任意バージョンへ切り替え可能
  - 子リポジトリ更新後、親が意図どおり参照更新できる
  - サブモジュール更新時の差分解釈が説明できる

## 7. 初回セッション用チェックポイント

- まずこれを実行:
  - `git status`
  - `git submodule status`
- まだサブモジュールが存在しない点を確認したうえで、Phase 1 を開始
- 不明点（コマンドの失敗時は、次の順で切り分け）:
  1. 親リポジトリの状態
  2. 子リポジトリの状態
  3. リモート参照
  4. `.gitmodules` と `.git/config` の不一致
